# 架构设计文档

## 整体架构

LegendServer采用分层架构设计，从上到下分为：

### 1. 网关层 (Gateway Layer)

网关服务器负责：
- 客户端连接管理
- 消息路由与转发
- 负载均衡
- 协议解析

### 2. 游戏逻辑层 (Game Logic Layer)

游戏服务器包含多个业务模块：
- 登录模块：账号认证、Token管理
- 角色模块：角色创建、属性管理
- 战斗模块：战斗计算、技能系统
- 聊天模块：频道管理、消息广播
- 好友模块：好友关系、在线状态
- 组队模块：队伍管理、组队匹配

### 3. 数据层 (Data Layer)

- MySQL：持久化存储
- Redis：缓存与会话存储
- 日志系统：异步日志记录

## 网络架构

### Reactor模式

```
┌─────────────────────────────────────┐
│            Main Reactor             │
│  - 监听端口                          │
│  - 接受新连接                        │
│  - 分发连接到Sub Reactor            │
└─────────────────────────────────────┘
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
┌───────────┐ ┌───────────┐ ┌───────────┐
│Sub Reactor│ │Sub Reactor│ │Sub Reactor│
│ EventLoop │ │ EventLoop │ │ EventLoop │
│  epoll    │ │  epoll    │ │  epoll    │
└───────────┘ └───────────┘ └───────────┘
```

### 消息流程

1. 客户端发送消息到网关
2. 网关解析协议头，获取消息ID
3. 根据消息ID路由到对应游戏服务器
4. 游戏服务器处理业务逻辑
5. 响应消息原路返回

## 多线程模型

### 线程分工

| 线程类型 | 数量 | 职责 |
|---------|------|------|
| Main Thread | 1 | 接受连接、负载均衡 |
| IO Thread | N | 处理网络IO |
| Logic Thread | M | 业务逻辑处理 |
| DB Thread | 1 | 数据库操作 |
| Log Thread | 1 | 日志写入 |

### 线程安全

- 使用互斥锁保护共享资源
- 任务队列实现生产者-消费者模式
- EventLoop的runInLoop实现跨线程调用

## Lua脚本引擎

### 设计目标

- 业务逻辑脚本化
- 支持热更新
- C++核心引擎提供高性能

### C++/Lua交互

```
┌──────────────┐     ┌──────────────┐
│   C++ Core   │────▶│   Lua VM     │
│  - 注册类     │     │  - 业务逻辑   │
│  - 注册函数   │◀────│  - 配置管理   │
└──────────────┘     └──────────────┘
```

## 数据库架构

### MySQL分库分表

- 按玩家ID哈希分库
- 日志表按时间分表
- 使用连接池管理连接

### Redis缓存策略

- 热点数据缓存
- 会话Token存储
- 在线状态维护
- 排行榜数据
